# 形式化验证方案（设计稿）

本文给出 CrabMate 在现有测试体系之上引入“轻量到中量”形式化验证的落地方案，目标是优先覆盖 **SSE 协议、状态机与关键不变量**，并逐步扩展到高风险路径。

## 1. 背景与问题

当前项目已有单元测试、金样回归、E2E 与 pre-commit 门禁，但仍存在以下风险：

- 协议演进时，后端分类器与前端分发分支可能发生“语义漂移”。
- 某些问题属于“路径组合爆炸”（例如状态机事件顺序），样例测试难以穷尽。
- 关键约束（如 stop/handled/plain 优先级）若只靠人工约定，长期维护成本高。

## 2. 目标与非目标

### 2.1 目标

- 对 SSE 控制面建立可执行不变量，避免契约隐式化。
- 通过属性测试提升组合路径覆盖率，降低回归漏检。
- 对关键状态机引入模型级验证（先离线、后可选 CI），覆盖“顺序与可达性”问题。
- 形成“代码-测试-文档”同源约束，减少跨端演进偏差。

### 2.2 非目标

- 不追求一次性把全仓库迁移到定理证明（如 Coq/Lean）。
- 不在第一阶段强制所有模块接入模型检查。
- 不替代现有 E2E/集成测试，只做高风险逻辑的加强层。

## 3. 验证对象分层

按风险与收益分三层推进：

1. **L1（已落地优先）**：SSE 协议与控制分类（后端协议 crate + 前端分发顺序）。
2. **L2（近期）**：会话流终止状态与取消/冲突路径（`StreamEndReason` 相关链路）。
3. **L3（中期）**：审批与工具执行编排状态机（请求、批准、拒绝、重试、收尾）。

## 4. 方法组合

采用“分层组合”而非单一工具：

- **属性测试（proptest）**：覆盖输入空间与顺序组合。
- **金样回归（fixtures）**：锁定跨端契约与历史行为。
- **跨端顺序快照测试**：检测前后端关键分支顺序漂移。
- **模型检查（TLA+/PlusCal）**：验证状态机安全性/活性约束（先离线执行）。

## 5. SSE 协议不变量（基线）

以下不变量必须长期保持：

- `error != null` 且 `code` 非空白时，分类必须为 `stop`，并优先于其他 handled 键。
- `tool_call.summary | arguments_preview | arguments` 任一存在且非空时，分类为 `handled`。
- `stream_ended.reason` 必须属于受控枚举集合，并可跨端一致解析。
- 金样中的控制面样本，Rust 共享分类器与前端分发分类必须一致。
- 前端关键控制分支的相对顺序不得反转（快照守护）。

## 6. 阶段化落地计划

### M0（已完成）

- 引入 `proptest`，为 `src/sse/protocol.rs` 增加属性测试。
- 为 `crabmate-sse-protocol/control_classify.rs` 增加属性测试。
- 增加前端 `sse_dispatch` 关键分支顺序快照测试。
- 提供一键回归脚本：`./scripts/check-sse-protocol.sh`。

### M1（1-2 周）

- 新增 `StreamEndReason` 链路不变量测试（队列收尾、行分类、前端消费一致性）。
- 在 `docs/SSE协议.md` 增加“必须不变量”小节，形成文档化契约。

### M2（2-4 周）

- 对审批/执行编排建立 TLA+ 模型（状态：Pending/Approved/Denied/Running/Ended）。
- 校验至少两个性质：
  - 安全性：禁止非法跃迁（如未审批直接执行敏感命令）。
  - 活性：已批准任务最终进入结束态（成功/失败/取消之一）。

### M3（持续）

- 将模型检查纳入可选 CI 任务（夜间或手动触发），失败时阻断高风险改动合并。
- 建立“新增协议键必须新增不变量测试”的评审清单。

## 7. CI 与门禁策略

默认门禁（提交前）：

- `pre-commit run --all-files`
- `./scripts/check-sse-protocol.sh`

可选增强门禁（分阶段引入）：

- `cargo test golden_sse_control --workspace`
- 模型检查任务（TLA+）作为非阻塞告警，稳定后升级为阻塞。

## 8. 目录与工件规划

- 代码测试：`src/sse/protocol.rs`、`crates/crabmate-sse-protocol/control_classify.rs`
- 契约金样：`fixtures/sse_control_golden.jsonl`
- 一键脚本：`scripts/check-sse-protocol.sh`
- 设计文档：`docs/形式化验证计划.md`（本文）
- （后续）模型文件建议放置：`specs/tla+/`（如 `sse_control.tla`）

## 9. 风险与缓解

- **风险：测试执行时间上升。**  
  缓解：属性测试样本数分层配置；本地快速、CI 全量。

- **风险：快照测试脆弱。**  
  缓解：仅快照“关键检查点”而非整文件文本。

- **风险：模型与实现脱节。**  
  缓解：模型评审必须关联具体代码路径与不变量编号。

## 10. 验收标准（Definition of Done）

满足以下条件视为方案落地有效：

- SSE 协议相关改动若破坏不变量，能在本地或 CI 稳定复现红测。
- 跨端（Rust 分类器 vs 前端分发）漂移可被自动检测。
- 文档、测试与脚本形成闭环，团队成员可按文档独立执行与扩展。
