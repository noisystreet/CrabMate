# 测试与质量检查

本文汇总在仓库根目录（或注明子目录）执行的**自动化测试与常用检查命令**，覆盖 Rust 后端、Leptos 前端、浏览器 E2E 与依赖审计。更细的模块与协议说明见 [`开发文档.md`](开发文档.md)。**`crabmate bench` 测评子系统的功能规划与专用测试策略**见 [`基准测试规划.md`](基准测试规划.md)（与本文通用测试清单分离）。**任务集如何设计与运维**（覆盖、可复现、成本、CI 分层）见 [`评测任务集设计.md`](评测任务集设计.md)。HumanEval 外挂判分脚本的**本地冒烟**（不调用 LLM）：`python3 scripts/humaneval_score_benchmark_results.py --tasks fixtures/benchmark/humaneval_tiny_tasks.jsonl --results fixtures/benchmark/humaneval_tiny_results.jsonl --output /tmp/tiny_scores.jsonl`（会执行 fixture 中的补全代码，见 `基准测试规划.md` §5.3）。

## 前置条件

- **Rust**：1.85+（edition 2024），见 [`README.md`](../README.md)。
- **端到端（E2E）**：Node.js、npm；首次需安装 Playwright 的 Chromium。
- **Web 静态资源**：E2E 与 `serve` 需要 **`frontend/dist/index.html`**，须先 **`cd frontend && trunk build`**（发布体积用 **`trunk build --release`**）。

## GitHub Actions（主 CI）

对 **`main`** 的 push / pull_request 会运行 [`.github/workflows/ci.yml`](../.github/workflows/ci.yml)：

- **`check-clippy-test`**：**`cargo check`**、**`cargo clippy`**（**`-D warnings`**）、**`cargo test --workspace --all-features`**
- **`e2e`**：在 **`frontend/dist`** 缓存未命中时执行 **`trunk build`**，再 **`cargo build --release -p crabmate`**，于 **`e2e/`** 跑 Playwright（**`CM_E2E_FIXTURES=1`**，不调用真实 LLM）；失败时上传 **`playwright-report`** 构件

圈复杂度、依赖审计、覆盖率等见其它 workflow（**`code-complexity.yml`**、**`dependency-security.yml`**、**`code-coverage.yml`**）。

## 提交前检查（pre-commit）

与 [`.pre-commit-config.yaml`](../.pre-commit-config.yaml) 对齐，建议在提交前执行：

```bash
pre-commit run --all-files
```

其中包含（节选）：

- **`cargo fmt --all`**
- **`cargo clippy --all-targets --all-features -- -D warnings`**
- **`frontend-wasm-check`** / **`frontend-clippy`**：暂存含 **`frontend/`** 时分别执行 **`cd frontend && cargo check --target wasm32-unknown-unknown`** 与 **`cd frontend && cargo clippy --all-targets --all-features -- -D warnings`**
- **`lizard-rust`**：Rust 圈复杂度（需 **`pip install lizard`**；**`scripts/lizard-rust.sh`** / **`scripts/lizard_rust_metrics.py`**：单函数 CCN 硬上限 **15**）
- **`fn-param-ratchet`**：Rust 函数形参个数（**`scripts/fn-param-ratchet.sh`** / **`scripts/fn_param_rust_metrics.py`**；硬上限 **32** 与 `scripts/fn_param_*.txt` 基线写死在 Python 内）
- **`fn-nloc-ratchet`**：Rust 函数体 **`nloc`**（lizard）与 **`.rs` 源文件物理行数**（同一脚本 **`scripts/fn-nloc-ratchet.sh`** / **`scripts/fn_nloc_rust_metrics.py`**，基线路径与写回策略写死在 Python 内）；函数棘轮 **`scripts/fn_nloc_max_baseline.txt`**、**`scripts/fn_nloc_top10_sum_baseline.txt`**；文件棘轮 **`scripts/rust_file_max_lines_baseline.txt`**、**`scripts/rust_file_top10_lines_sum_baseline.txt`**；与 **`lizard-rust`** 同在 CI **`.github/workflows/code-complexity.yml`**
- **覆盖率**：**`.github/workflows/code-coverage.yml`** 默认仅 **Actions 手动触发**（`workflow_dispatch`）；本地仍可 `cargo llvm-cov` + **`scripts/check_coverage_ratchet.py`**
- **`cargo test golden_sse_control`**（当改动 `fixtures/sse_control_golden.jsonl`、`crates/crabmate-sse-protocol/control_classify.rs` 或 `frontend/src/sse_dispatch/` 下任意文件时由钩子条件触发）

未安装 pre-commit 时，可至少执行：

```bash
cargo fmt --all
cargo clippy --all-targets --all-features -- -D warnings
cd frontend && cargo clippy --all-targets --all-features -- -D warnings
bash scripts/lizard-rust.sh
bash scripts/fn-param-ratchet.sh
bash scripts/fn-nloc-ratchet.sh
```

**说明**：`pre-commit run --all-files` **不会**跑 `commit-msg`；提交说明格式在 **`git commit`** 时由 conventional-pre-commit 校验（见 [`.cursor/rules/conventional-commits.mdc`](../.cursor/rules/conventional-commits.mdc)）。

## Rust：工作区单元测试与集成测试

在**仓库根**执行（默认覆盖 workspace 成员：`crabmate`、`crabmate-web`、`crabmate-sse-protocol`）：

```bash
cargo test
```

### 按包筛选

| 包 | 命令 | 说明 |
| --- | --- | --- |
| 主程序与后端库 | `cargo test -p crabmate` | 大部分 `src/` 与 `tests/` 测例在此包 |
| SSE 协议 crate | `cargo test -p crabmate-sse-protocol` | 协议版本与文档标记自检等 |
| Web 前端 crate | `cargo test -p crabmate-web` | 见下文「前端（Leptos）」 |

### 按名称过滤（示例）

```bash
cargo test golden_sse_control
cargo test -p crabmate-sse-protocol golden_sse_control
cargo test tool_result_envelope_golden
```

### SSE 协议一键回归脚本

仓库提供脚本：

```bash
./scripts/check-sse-protocol.sh
```

该脚本会串行执行：

- `cargo test -p crabmate sse::protocol::tests`
- `cargo test -p crabmate-sse-protocol control_classify`

如需代理拉取依赖，可先设置：

```bash
export http_proxy=http://localhost:8118
export https_proxy=http://localhost:8118
./scripts/check-sse-protocol.sh
```

改动 **SSE 控制面**分支顺序时，须同步金样并跑 `golden_sse_control`（见 [`SSE协议.md`](SSE协议.md)）。**合并/发版前**若改动跨 crate 或公共 API，建议全量 **`cargo test`**（与 [`.cursor/rules/rust-clippy-and-tests.mdc`](../.cursor/rules/rust-clippy-and-tests.mdc) 一致）。

### 可选：nightly 测试

部分环境会用 nightly 跑全量：

```bash
cargo +nightly test
```

## 前端（Leptos / `frontend`）

### 宿主目标单元测试（默认）

在仓库根：

```bash
cargo test -p crabmate-web
```

或在 `frontend` 目录：

```bash
cd frontend && cargo test
```

覆盖 Markdown 净化、会话逻辑、`debounce_schedule` 等纯 Rust 逻辑（不启动浏览器）。

### WASM 目标测试（可选）

`wasm-bindgen-test` 用例需安装与 **`Cargo.lock`** 中 **`wasm-bindgen`** 版本一致的 **`wasm-bindgen-cli`**，并指定 test runner。当前锁文件中为 **0.2.114**，示例：

```bash
cargo install wasm-bindgen-cli --version 0.2.114 --locked
CARGO_TARGET_WASM32_UNKNOWN_UNKNOWN_RUNNER=wasm-bindgen-test-runner \
  cargo test --target wasm32-unknown-unknown -p crabmate-web
```

若锁文件升级了 `wasm-bindgen`，请将安装命令中的版本号改为锁文件中的版本。

### 编译检查（无测例）

协议或前端大改时，至少做一次 WASM 目标类型检查：

```bash
cd frontend && cargo check --target wasm32-unknown-unknown
```

### 构建静态包（非测试，但 E2E / serve 依赖）

```bash
cd frontend && trunk build
# 发布或对齐生产体积：
# cd frontend && trunk build --release
```

## 浏览器端到端（E2E，Playwright）

目录：**`e2e/`**。对 **`POST /chat/stream`**、**`/workspace`** 等使用 **route 桩**，**不调用真实 LLM**。用例含 **`smoke`**、**`user-data`**、分页、**`sse-control`** / **`sse-clarification`** / **`sse-staged-plan`** / **`sse-approval-actions`**、**`prefs-*`**、**`settings-appearance`** / **`settings-llm`**、**`keyboard-shortcuts`**、**`session-list`** / **`session-crud`**、**`status-bar`**。须 **`CM_E2E_FIXTURES=1`**（由 **`playwright.config.ts`** 注入）。公共辅助 **`e2e/tests/helpers/`**（**`settings`**、**`sendStubMessage`** 等）。本地一键：**`scripts/e2e.sh`**。选择器优先 **`data-testid`**。

```bash
cd frontend && trunk build
cd ../e2e && npm ci
npx playwright install chromium
npm test
```

说明：

- **`playwright.config.ts`** 会启动 **`cargo run -- serve --port 18081`** 并等待 **`GET /health`**。
- 端口可通过环境变量 **`E2E_PORT`** 覆盖，例如：`E2E_PORT=19090 npm test`。
- 本地非 CI 时可能**复用**已在该端口运行的 `serve`（见配置中的 `reuseExistingServer`）。
- 调试：`cd e2e && npm run test:ui`。
- 脚本：`./scripts/e2e.sh`（先 `trunk build`，有 release 二进制时设 `CI=true` 加速 webServer）。

Linux 上若 `cargo` 在 **wayland** 相关依赖处失败，见 [`开发文档.md`](开发文档.md) § E2E 中的 **`libwayland-dev`** 说明。

## 依赖安全与许可证（与 CI 对齐）

工作流见 [`.github/workflows/dependency-security.yml`](../.github/workflows/dependency-security.yml)。本地需安装 **`cargo-audit`**、**`cargo-deny`** 后执行：

```bash
cargo audit
cargo deny check licenses bans sources
```

配置见根目录 **`deny.toml`**。此类检查**未**接入 pre-commit，避免每次提交拉取 advisory 数据库。
