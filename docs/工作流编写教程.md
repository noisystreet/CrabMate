# 工作流编写教程

**受众**：在 CrabMate 工作区内编写、校验与运行 DAG 工作流的使用者与维护者。  
**前置**：已配置工作区（Web 侧栏、`--workspace` 或 `run_command_working_dir`）。  
**关联**：[docs/工具说明.md](工具说明.md)（`workflow_execute`）、[docs/命令行与路由.md](命令行与路由.md)（`workflow` 子命令）、[docs/工作流Markdown作者层设计.md](工作流Markdown作者层设计.md)（设计稿）。

---

## 1. 工作流是什么

工作流把多个**内置工具**按 **DAG**（有向无环图）编排：可并行、可串行、可按前驱成功/失败分支、可对列表逐项执行。执行入口：

| 方式 | 说明 |
|------|------|
| **CLI** | `crabmate workflow validate` / `compile` / `run <文件>` |
| **Agent** | 工具 `workflow_execute`，或参数 **`workflow_file`** 指向工作区内 YAML/MD |
| **诊断** | `crabmate doctor` 会扫描工作区 `.crabmate/workflows/*.{yaml,yml,md}` |

**不要求 `API_KEY`**：`workflow` 子命令只做编译与工具执行，不调用大模型。

---

## 2. 文件放哪里

**本仓库示例**（仓库根目录，可直接 `workflow validate` / `run`）：

```text
examples/workflows/
  ci.yaml
  serial_check.yaml
  README.md
  …
```

在**自己的项目工作区**里运行时，可把 YAML 复制到例如 **`.crabmate/workflows/`**，再通过 `workflow_file` 引用（路径相对工作区根）。

回归测试夹具在 **`fixtures/workflows/`**（维护者用，不必复制）。

---

## 3. 两种编写模式（二选一）

| 模式 | 适用 | 根级形态 |
|------|------|----------|
| **steps** | 人类可读、易写分支/循环 | `version: 2` + `steps[]`（`after` / `when` / `for_each` / `repeat`） |
| **nodes** | 已是 DAG、或只用内置模板 | `version: 2` + `workflow.nodes` 或 `workflow.workflow_template` |

**不能**在同一文件里同时写 `steps` 与 `workflow.nodes`。

两种模式均须根级 **`version: 2`**（整数）。加载时做 [JSON Schema](../fixtures/workflows/schema/) 校验；未知顶层字段会被拒绝。

## 4. 快速开始

### 4.1 用内置 CI 模板（nodes 模式）

`examples/workflows/ci.yaml`：

```yaml
version: 2
workflow:
  fail_fast: true
  workflow_template: rust_ci_light
```

`rust_ci_light` 展开为：`cargo_fmt_check` → `cargo_check` → `cargo_clippy` → `cargo_test`。

```bash
cargo run -- workflow validate examples/workflows/ci.yaml
cargo run -- workflow run examples/workflows/ci.yaml
```

其它模板：`code_review`、`refactor_precheck`（后者须 `refactor_symbol`）。见 [工具说明](工具说明.md)。

### 4.2 手写串行检查（steps 模式）

`examples/workflows/serial_check.yaml`：

```yaml
version: 2
workflow:
  fail_fast: true
steps:
  - id: diff
    tool: git_diff
    args: { mode: all }
  - id: clippy
    tool: cargo_clippy
    after: [diff]
    args: { all_targets: true }
  - id: test
    tool: cargo_test
    after: [clippy]
    args: {}
```

- **`after`**：编译为节点的 **`deps`**，表示「等谁跑完再跑我」。
- **`id`**：节点唯一标识，供 `after` / `when` / 占位符引用。

```bash
cargo run -- workflow validate --json examples/workflows/serial_check.yaml
```

输出含 `author_spec_version`、`author_mode`、`execution_layers` 等。

### 4.3 Markdown 围栏（文档即工作流）

`examples/workflows/ci_from_markdown.md` 中：正文随意写；另起一行用 **\`\`\`crabmate-workflow** 围栏包住 YAML（见该文件原文）。围栏内示例：

```yaml
version: 2
workflow:
  fail_fast: true
  workflow_template: rust_ci_light
```

围栏标识须为 **`crabmate-workflow`**（大小写敏感）。CLI 对 `.md` 会提取**首个**围栏块。

## 5. steps 模式：常用能力

### 5.1 串行：`after`

```yaml
- id: b
  tool: cargo_clippy
  after: [a]
```

### 5.2 分支：`when` + `branch`

仅当前驱 **失败** 时跑测试：

```yaml
- id: clippy
  tool: cargo_clippy
  args: { all_targets: true }

- id: test_on_fail
  tool: cargo_test
  after: [clippy]
  when:
    from: clippy
    branch: failure
```

`branch` 取值：`success` | `failure`。未满足条件的节点记为 **skipped**（可在 trace 中看到）。

### 5.3 多路分支：`when` + `match`

前驱输出须为可解析 JSON，且含匹配字段（如 `branch`）：

```yaml
- id: route
  tool: diagnostic_summary
  args: {}

- id: hotfix
  tool: git_diff
  after: [route]
  when:
    from: route
    match:
      field: branch
      equals: hotfix
```

也可用 **`in: [skip, none]`** 代替 `equals`。

### 5.4 逐项：`for_each`

对前驱工具输出中的列表逐项调用（**有界**）：

```yaml
- id: diff
  tool: git_diff
  args: { mode: all }

- id: outline_each
  tool: rust_file_outline
  after: [diff]
  for_each:
    from: diff
    json_path: "/changed_rs_paths"
    item_var: path
    max_items: 10
  args:
    path: "{{path}}"
```

- **`max_items`**：硬上限（编译 clamp ≤ 64）。
- **`static_items`**：无运行时 JSON 时可在编译期展开（见示例 `for_each_static.yaml`）。
- 同一步**不能**与 `repeat` 同时使用。

### 5.5 有界重试：`repeat`

最多 3 次 `cargo_test`，**任一轮成功则不再生成后续节点**：

```yaml
- id: flaky
  tool: cargo_test
  repeat:
    count: 3
    stop_on: success   # 默认 success；never 表示跑满 count
  args: {}
```

编译为 `flaky_1` → `flaky_2` → `flaky_3` 链式依赖，**不是** `while` 循环。

### 5.6 语法糖：`kind: choice`

把多分支收拢为一个 choice 步（编译期展平为带 `when` 的 steps），见 `examples/workflows/choice_lint.yaml`。

## 6. nodes 模式

直接写 DAG（与 `workflow_execute` 的 `workflow.nodes` 同形）：

```yaml
version: 2
workflow:
  fail_fast: false
  max_parallelism: 4
  nodes:
    - id: check
      tool: cargo_clippy
      tool_args:
        all_targets: true
      deps: []
    - id: test
      tool: cargo_test
      tool_args: {}
      deps: [check]
```

字段对应：`tool` / `tool_name`、`tool_args` / `args`、`deps`（等同 steps 的 `after`）。

---

## 7. 字段速查（steps 单步）

| 字段 | 必填 | 说明 |
|------|------|------|
| `id` | 是* | *choice 步除外 |
| `tool` / `tool_name` | 二选一 | 须为内置工具名 |
| `args` / `tool_args` | 否 | 工具参数；执行前另有工具 Schema 校验 |
| `after` | 否 | 依赖节点 id 列表 |
| `when` | 否 | `branch` 或 `match` |
| `for_each` | 否 | 须 `from`、`max_items`、`json_path` 或 `static_items` |
| `repeat` | 否 | `count` 1–5，`stop_on` |
| `label` | 否 | 仅展示 |
| `requires_approval` | 否 | 人工审批节点 |
| `max_retries` | 否 | 0–5，可重试类失败 |

`workflow` 元数据：`fail_fast`、`max_parallelism`、`compensate_on_failure` 等。

完整 Schema：`fixtures/workflows/schema/workflow_author_v2_steps.schema.json`、`workflow_author_v2_nodes.schema.json`。

## 8. CLI 命令

```bash
# 编译为 workflow_execute 用的 JSON（stdout）
cargo run -- workflow compile examples/workflows/serial_check.yaml

# 校验：作者层 schema + DAG + 工具必填参数
cargo run -- workflow validate examples/workflows/serial_check.yaml
cargo run -- workflow validate --json examples/workflows/serial_check.yaml

# 执行（等同 workflow_execute + workflow_file）
cargo run -- workflow run examples/workflows/ci.yaml

# 工作区诊断（含 workflows 目录）
cargo run -- doctor
```

全局加 **`--workspace /path/to/repo`** 若不在仓库根执行。

---

## 9. 在对话里使用

`workflow_execute` 参数示例：

```json
{
  "workflow_file": "examples/workflows/serial_check.yaml"
}
```

路径相对**工作区根**；若使用本仓库 `examples/workflows/`，请以仓库根为工作区，或复制到项目工作区后再引用。

或内联（无 `version` 的纯 `workflow.nodes` 仍兼容；含 **`steps`** 时须 **`version: 2`**）：

```json
{
  "workflow": {
    "fail_fast": true,
    "nodes": [
      { "id": "a", "tool_name": "diagnostic_summary", "tool_args": {}, "deps": [] }
    ]
  }
}
```

仅校验、不执行：在 `workflow` 中加 **`"validate_only": true`**。

## 11. 常见问题

| 现象 | 处理 |
|------|------|
| `缺少 version` | 根级加 `version: 2` |
| `不能同时包含 steps 与 nodes` | 只保留一种模式 |
| `与 schema 不一致` | 检查是否多写了未定义字段；对照 `fixtures/workflows/schema/` |
| `未知 workflow_template` | 仅用 `rust_ci_light` / `code_review` / `refactor_precheck` |
| `after 引用未知 id` | 检查 id 拼写；`for_each` 的 base_id 也可被 `after` 引用 |
| 工具参数报错 | `workflow validate` 通过后仍可能因工具 Schema 失败；对照 [工具说明](工具说明.md) |

**不支持**：无界 `while`、块内任意脚本、单轮 DAG 内「直到全绿」——请用多轮对话或每轮小工作流。
