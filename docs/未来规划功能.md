## Web 身份与账号体系（不在 CrabMate 进程内实现）

**共识**：**不在 CrabMate 内做按用户账号体系**（注册登录、JWT 会话、按 `user_id` 隔离会话存储等）。进程内保持 **`web_api_bearer_token`** / **`CM_WEB_API_BEARER_TOKEN`** 的**共享密钥**模型：`Authorization: Bearer …` 或 `X-API-Key …` 与配置值做常量时间比对，通过即视为「已授权调用方」，**不区分自然人**。实现见 **`src/web/chat_handlers/auth.rs`**。

**推荐部署形态**：在 CrabMate **前面**放置已具备账号体系的 **API 网关、反向代理或 BFF**（OAuth2/OIDC、Keycloak、Authentik、云厂商 API 网关等），由网关完成登录、配额、审计；再向 CrabMate 注入凭证：

1. **模式 A（常见）**：网关校验终端用户后，为下游请求附上与本进程配置的 **`web_api_bearer_token` 相同的 Bearer**（或由 BFF 持有该机密并代发）。CrabMate 仍只见「单一服务级密钥」；**租户 / 用户隔离**在网关与上游业务系统中完成。适合「每租户独立部署 CrabMate」或「BFF 统一代填机密」。
2. **模式 B**：网关终止 TLS 与用户会话后，内网到 CrabMate 使用**固定共享密钥**，并由网关写入 **`X-User-Id` 等头**表达身份。**注意**：CrabMate **当前不消费**此类信任头做授权；若自行扩展，必须保证**仅可信内网**且请求**无法绕过网关**伪造头，否则等同后门。

**与上游 LLM 密钥**：环境变量 **`API_KEY`**（或 Web **`client_llm.api_key`**）仅用于调用 **`chat/completions`** 等上游，与「谁在用 CrabMate HTTP API」是两条线；网关侧可按租户转发不同上游密钥属于网关/BFF 职责。

**可选演进（仍非「进程内账号」）**：若将来在仓库内做**极小**改动，可考虑「多枚服务级 API Key → 租户 id 映射」等，仍与完整账号体系不同；若需每用户会话落库隔离，应优先在 **BFF + 独立存储** 或 **多实例** 上解决，而非在 Agent 核心内复制 IdP。

**相关文档**：**`docs/design/web_api_integration.md`**（桥接、多租户与网关分工）、**`docs/配置说明.md`**（Web API 鉴权与热重载边界）。

## 观众角色（编排层侧向点评）
