# Agent 评测任务集设计

**状态**：设计说明（规范任务集建设原则与运维方式，**不**绑定具体商业数据集采购）。**受众**：维护者、评测负责人、产品与 CI 配置者。  
**语言**：中文。  
**关联文档**：**`docs/基准测试规划.md`**（`crabmate bench` 子系统与 adapter 细节）、**`benchmark/README.md`**（快速跑法）、**`docs/规划执行验证架构.md`**（P-E-V、`acceptance`）、**`docs/测试指南.md`**（通用测试与 E2E 边界）。

---

## 1. 文档目的

把「自动测 agent 完成真实任务的能力，并收集执行信息用于改进」落到**可维护的任务集**上，避免：

- 判据含糊、无法回归；
- 环境漂移导致指标不可比；
- 成本与密钥管理失控；
- 与产品能力（分阶段规划、工具白名单、工作流等）脱节。

本文定义**任务集层面的设计约束**；JSONL 字段与 adapter 行为仍以 **`src/runtime/benchmark/types.rs`** 与 **`docs/基准测试规划.md`** 为准。

---

## 2. 设计原则总览

| 维度 | 要求（摘要） |
|------|----------------|
| **可测性** | 优先机器可判定成功（退出码、测试输出、文件存在、结构化 JSON）；少用纯 LLM 当唯一裁判。 |
| **可复现** | 固定工作区快照（commit、归档或最小 fixture 仓）；声明 OS/工具链依赖；接受模型方差，必要时 **k 次重复** 报告成功率区间。 |
| **安全** | 任务不诱导越权读密钥、乱连外网；需网络时用桩或固定 allowlist；日志与产物脱敏。 |
| **成本** | 每任务设 **轮次/工具/墙钟** 上限；PR 级 **smoke** 与 **nightly 全集** 分层。 |
| **对齐产品** | 显式覆盖要卖的能力：分阶段、`patch_planner`、`steps[].acceptance`、`workflow_execute`、多角色、`run_command` 审批等（用配置与任务描述组合触达）。 |
| **反泄漏 / 反作弊** | 答案与 hidden 断言分离；公开 benchmark 接受「可能见过题」——用变体或组合子任务缓解。 |
| **可维护** | 稳定 `task_id`、标签、**最小 CrabMate 版本**、废弃标记；失败包保留诊断字段。 |
| **公平对比** | 多模型/多配置对比时 **单因素变化**，其余钉死。 |

## 3. 任务分层与覆盖矩阵

建议用二维表驱动选题，再落到具体 JSONL：

| 难度 | 典型能力 | 判据类型 |
|------|----------|----------|
| **L0** | 单轮只读工具、短回答 | 输出子串 / 文件存在 |
| **L1** | 多工具顺序、简单补丁 | `cargo check` 或脚本退出码 |
| **L2** | 规划—执行—校验闭环 | `steps[].acceptance` 或 bench 外挂判分（如 HumanEval 路径，见 **`基准测试规划.md`** §5） |
| **L3** | 长上下文、工作流 DAG、失败恢复 | 组合 L1+L2 + 超时与重试统计 |

**覆盖维度（标签示例）**：`readonly` / `patch` / `test_runner` / `workflow` / `staged_plan` / `http_stub` / `mcp`（若启用）。

## 4. 任务元数据（建议字段）

在 **`BenchmarkTask`** 已有字段之上，建议在任务仓库或 JSONL 旁维护**扩展元数据**（独立 JSON 或 JSONL 注释字段，以 adapter 是否透传为准；未透传则放在侧车文件 `tasks_meta.json`）：

| 字段 | 用途 |
|------|------|
| `task_id` | 稳定主键，与 `benchmark_results.jsonl` 对齐。 |
| `version` | 任务正文或 fixture 版本，变更判分规则时递增。 |
| `tags[]` | 筛选与报表聚合。 |
| `min_crabmate_version` | 避免旧二进制跑新语义任务。 |
| `deprecated` | 软下线，CI 默认跳过。 |
| `max_tool_rounds` / `timeout_secs` | 与 bench CLI 一致或更严（取 min）。 |
| `expected_artifacts[]` | 可选：列出应生成的路径模式，供外挂校验。 |

## 5. 执行信息采集（用于改进）

| 来源 | 内容 | 用途 |
|------|------|------|
| **`benchmark_results.jsonl`** | 每任务一行结构化结果 | 聚合成功率、耗时、失败阶段 |
| **`save-session` / Web 导出 JSON** | 完整消息与工具轨迹 | 沉淀新任务、做 diff 回归 |
| **`tool-replay`** | 无 LLM 重放工具序列 | 工具层回归，与模型解耦 |
| **进程日志** | `RUST_LOG`、`--log` | 排障与成本粗估 |
| **`GET /status`** | 队列、`per_active_jobs`、管道计数等 | 运行态与 **`plan_rewrite`** / **分阶段补丁** 计数对照（见 **`docs/开发文档.md`**） |
| **可选 `thinking_trace`** | 推理/阶段轨迹 | 细粒度调参（注意体积与脱敏） |

**聚合指标建议**：按 `model` × `tags` × 配置键（如 `staged_plan_feedback_mode`）统计——成功率、平均轮数、`plan_rewrite` 耗尽率、补丁轮用尽率、高频 `error_code`。

## 6. CI 与运行策略

| 层级 | 内容 | 频率 |
|------|------|------|
| **契约** | JSONL 解析、`validate_task`、fixture 冒烟 | **pre-commit / 每 PR** |
| **Smoke bench** | 极小 `generic` 或单行任务，**可不调用 LLM** 的路径优先 | **每 PR**（若未来引入 mock）或 **每日** |
| **真模型子集** | 固定密钥于 CI Secret，小任务集 + 硬超时 | **nightly**；**不**阻塞默认 PR |

密钥与响应体**不得**写入仓库或公开报告全文；CI 日志做截断与占位符。

## 7. 与现有子系统的关系

```text
任务集 JSONL / fixture 仓库
        │
        ├─► crabmate bench（见 基准测试规划.md）
        │       └─► benchmark_results.jsonl → 聚合 / 看板
        │
        ├─► tool-replay（从会话或手写 fixture，无 LLM）
        │
        └─► 会话导出 → 筛选 → 新任务 / 回归用例
```

---

## 8. 演进与评审

- **新增任务**：需至少一人审「判据是否硬、环境是否钉、是否安全、是否可重复跑」。
- **废弃任务**：`deprecated=true` 后保留 **90 天**或一个发布周期再删，避免历史曲线断裂。
- **变更判据**：递增 `version`，并在结果文件中写入版本，避免旧结果与新任务混读。
